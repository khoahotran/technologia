# Debug Changelog - 403 Forbidden Error on Add to Cart
**Session ID:** 20260303_104202_719288  
**Date:** 2026-03-03  
**Issue:** User receives 403 Forbidden when adding to cart immediately after login

## Root Cause Analysis

### Problem Identified
According to API_INTEGRATION_AUDIT_DOC.md (section 5.1):
- Backend returns `403 Forbidden` instead of `401 Unauthorized` when token is expired
- Frontend HTTP client immediately redirects to login on 403 status
- This prevents the token refresh mechanism from attempting to refresh the token
- Result: User gets "Access forbidden" error despite just logging in with valid credentials

### Code Flow Analysis
1. **Login (works fine):** User logs in → tokens stored in storage + cookies
2. **Add to Cart (fails):** 
   - Request includes `Bearer {token}` header
   - If token expired, backend returns 403
   - HTTP interceptor line 161-167 immediately redirects to `/login?error=forbidden`
   - **No token refresh attempt is made**

## Changes Made

### 1. Modified HTTP Response Interceptor (infrastructure/http/client.ts)
**File:** `infrastructure/http/client.ts`  
**Lines Changed:** 160-176

**What was changed:**
- Removed immediate redirect on 403 status
- Updated condition to treat both 401 AND 403 as potential token expiration
- Now attempts token refresh for both statuses before redirecting

**Before:**
```typescript
// Handle 403 Forbidden - Redirect to Login
if (status === HTTP_STATUS.FORBIDDEN) {
    httpLogger.error("Access forbidden - redirecting to login", { url: originalRequest?.url });
    if (typeof window !== "undefined") {
        window.location.href = "/login?error=forbidden";
    }
    return Promise.reject(new AuthenticationError("Access forbidden"));
}

// Handle 401 Unauthorized - Token Refresh
const isAuthRoute = originalRequest?.url?.includes("/auth/login") || ...;

if (status === HTTP_STATUS.UNAUTHORIZED && !originalRequest._retry && !isAuthRoute) {
```

**After:**
```typescript
// Handle 401 Unauthorized & 403 Forbidden - Token Refresh
// NOTE: Backend may return 403 instead of 401 for expired tokens
// So we treat both as token expiration and attempt refresh
const isAuthRoute = originalRequest?.url?.includes("/auth/login") || ...;

const shouldAttemptRefresh = (status === HTTP_STATUS.UNAUTHORIZED || status === HTTP_STATUS.FORBIDDEN) 
    && !originalRequest._retry 
    && !isAuthRoute;

if (shouldAttemptRefresh) {
```

**Why:** This allows the token refresh logic to run when receiving 403, matching the intended behavior described in the audit document.

### 2. Updated Token Refresh Error Handling (infrastructure/http/client.ts)
**File:** `infrastructure/http/client.ts`  
**Lines Changed:** 229-239

**What was changed:**
- Updated error redirect to preserve the original error status (403 vs 401)
- Improved error message clarity

**Before:**
```typescript
catch (refreshError) {
    processQueue(refreshError, null);
    authStorage.clearTokens();

    // Redirect to login page
    if (typeof window !== "undefined") {
        window.location.href = "/login?error=session_expired";
    }

    httpLogger.error("Token refresh failed", refreshError);
    return Promise.reject(new AuthenticationError("Session expired"));
}
```

**After:**
```typescript
catch (refreshError) {
    processQueue(refreshError, null);
    authStorage.clearTokens();

    // Redirect to login page with appropriate error
    const errorParam = status === HTTP_STATUS.FORBIDDEN ? "forbidden" : "session_expired";
    if (typeof window !== "undefined") {
        window.location.href = `/login?error=${errorParam}`;
    }

    httpLogger.error("Token refresh failed", refreshError);
    return Promise.reject(new AuthenticationError("Session expired or access forbidden"));
}
```

**Why:** Preserves error context for better user messaging and debugging.

## Expected Behavior After Fix

### New Flow When Backend Returns 403
1. ✅ User logs in successfully
2. ✅ User adds to cart
3. Request sent with Bearer token
4. Backend returns 403 (expired token)
5. **NOW:** Frontend attempts token refresh (new behavior)
6. **If refresh succeeds:** Request is retried with new token → Add to cart succeeds
7. **If refresh fails:** User is redirected to login with `error=forbidden`

### Backwards Compatibility
- 401 responses still work the same (token refresh attempted)
- Auth route responses (login, register, refresh-token) are still skipped
- No retry limit changes (still max 1 retry)

## Critical Finding: httpOnly Cookie Issue
**File:** `lib/api-route.ts` line 66

The access token is set with `httpOnly: true`:
```typescript
cookieStore.set(COOKIE_NAMES.ACCESS_TOKEN, accessToken, {
    httpOnly: true,  // ❌ JavaScript CANNOT read this!
});
```

**Impact:**
- httpOnly cookies are sent automatically by browser with requests ✅
- But JavaScript code CANNOT read them with `document.cookie` ❌
- However, `authStorage.getAccessToken()` tries to read from cookie first, which will return null
- Then it falls back to localStorage, which should work ✅

**Potential Issue:**
If localStorage token is missing but httpOnly cookie exists, the request will have the httpOnly cookie sent automatically by browser, BUT our code won't know about it and won't log it.
